Walkthrough

Shiny is a wonderful tool for building interactive web apps using python! R lovers may be familiar with the original Shiny package in R, but now Shiny works with python too! Shiny uses reactivity to update your application when users change different inputs, and provides numerous tools to build a beautiful Graphical User Interface (GUI) that can be used as a Dashboard, a Presentation, or even a Pedagodical activity! Shiny allows you to interface with all of your favorite python packages like pandas, seaborn, plotnine, sklearn, and keras!

Parts of an App

Shiny Apps have two major parts: the UI (which determines how the app should look) and the server (which determines how the app should behave).

UI

First, let’s start with a basic skeleton for a Shiny App. The app has both a UI component (stored in app_ui) which is using page_fluid to create a basic container for our UI. The UI object tells the app what it’s basic structure should be. We’ll also take advantage of the layout_sidebar, panel_sidebar, and panel_main functions which allow us to create a skeleton app that has two main panels: a sidebar (where all our interactive inputs will go), and a main panel (where our recommendations will go).

Server

Then it has a server function, which is going to tell Shiny how to dynamically update. This is where we’ll define any part of our app that might change based on user input. The server helps us define the computational graph for our app, which will allow shiny to know what parts of the app need to be updated when user input changes.

User Inputs

The whole point of Shiny apps are interactivity: we want our app to change in response to user input. So let’s add some places where users can give input. In Shiny for Python, you can add user inputs by adding ui.input_*() to the ui portion of the app (the * indicates that you can use all sorts of input types, like ui.input_slider() or ui.input_selectize()). We want THREE pieces of information from our users:

  1. What Item they want (so we can recommend similar items)
  2. What restaurants they want to see recs from
  3. How many recs they want

So let’s add three inputs to our app in the #UI section, specifically in the panel_sidebar. Each input function needs a name so that we can reference it later, so we’ll call our inputs choice (item to match), limits (restaurants allowed), and n (number of recommendations). Each input function also needs a label that will tell the user what the button is for. We’ll use the text "Select Item" (item to match), "Restaurants Allowed" (restarants allowed), and Number of Recommendations" (number of recommendations).

Selectize

choices

Our first two inputs, choice and limits both use ui.input_selectize() which creates a dropdown menu for our users. We need to tell input_selectize the options our users have to select from, in the form of a dictionary. The keys in the dictionary are the values that the input_selectize function will return to us to use. The values in the dictionary are what the user will see as their option. We’ll supply this dictionary to the choices argument in ui.input_selectize().

I’ve already defined these for you in the choices.py file, so we’ll go ahead and load those, and use them here (d for items, and rest for restaurants).

multiple

We also can tell ui.input_selectize() whether or not users are allowed to select multiple items from the dropdown menu. For our first input (choice), we only want one selection, so we’ll use the default value of multiple = False. BUT for the second input (limits, which tells us which restaurants to choose recs from), we want them to be able to select many. So we’ll use multiple = False.

selected

Lastly, we have the option of telling shiny which options should be selected before the user changes anything. The default is to select the first item, but we can override that. For our second input (limits, which tells us which restaurants to choose recs from), we’ll tell shiny to select ALL of them by default. We’ll get a list of the possible options by using list(rest.keys()) so we can provide ui.input_selectize() with a list of items to select by default.

Numeric

And we saved the easiest input for last! The last input will be a numeric input that allows the user to choose a number by typing or incrementing. We’ll use ui.input_numeric() for this. The four extra arguments we’ll supply are value, min, max, and step.

  • value tells shiny what the default value for this input should be (before the user changes anything)
  • min and max tell shiny what the lowest and highest value the user can choose should be
  • step tells shiny how the input should increment when the user clicks the up or down arrows
from shiny import App, render, ui, reactive
from pathlib import Path
import shinyswatch


# import dictionary from local file (to save space)
import choices
d = choices.all_items
rest = choices.restaurants

# UI

    # app title
    ui.h1("Fast Food Recommender", align = "center"),
    
    # sidebar
    ui.layout_sidebar(
        ui.panel_sidebar(
            # item to match
            ui.input_selectize("choice", "Select Item",
                               choices = d
                               ),
            # restaurants allowed                   
            ui.input_selectize("limits", "Restaurants Allowed",
                               choices = rest,
                                multiple = True,
                                selected = list(rest.keys()),
                            ),
            # number of recommendations
            ui.input_numeric("n", "Number of Recommendations", value = 3, min = 1, max = 50, step = 1),
            
        ),

        ui.panel_main(
           ui.h2("Recommendations"),
        ),
    ),
    
)

    pass

Nearest Neighbors Algorithm

We’ll skip over the details here, but we’re going to use the NearestNeighbors() algorithm from sklearn to make recommendations for fast food items people might want to eat. To do this, we need data! The csv fastfood_calories.csv should already be loaded into your app (if not, you can get it here and upload it using the upload button near the filename tabs at the top of your script).

Then we’ll pre-process (aka scale) our data and fit a NearestNeighbors() model (we’ll also load all the data science packages we need to do so). Because the dataset and model aren’t dynamic, we’re loading it once at the beginning of the script.

Server

That’s because we haven’t told shiny how to create this table. We need to do that in the server() function, which is where all the dynamic parts of our app live.

The only dynamic output we have is our table, so let’s add that. First, if we’re telling shiny how to render an output, we need the @output decorator. Then, we need to tell shiny what kind of output we’re rendering with another decorator: @render.table. The .table part tells shiny that we’re rendering a table. If we were rendering a plot, we’d use @render.plot().

Right below these two decorators, we define a function. This function should have the same name as the id of the output we’re rendering (table, here). The function should return the object to be rendered, but first we’ll need to create the table. We’ll do this in 5 easy steps.

  1. Grab the index of the item the user wants to see recs for (i = data.index[data['item'] == input.choice()])
  2. Use that index to grab all the recommendations for this item (recs = data.iloc[neighbors[i][0]])
  3. Filter the recommendations so it only includes restaurants allowed by the user (recs = recs.loc[recs["restaurant"].isin(input.limits()),])
  4. Get rid of the index column (recs = recs.iloc[: , 1:] # get rid of extraneous column)
  5. Select the first n recs (recs.iloc[0:(input.n()+1)])

Most of this is basic pandas code, but notice all those input.*() calls. These allow us to get the current values for our different user inputs. input.n() will grab the current value the user gave for the input n, input.choice() will grab the current item that the user wants to get recs for. They also help shiny figure out the computational graph for your app, because shiny can figure out which parts of the app should be re-rendered when inputs change, and which can stay the same. When an input changes, all the downstream nodes in the computational graph will update.

from shiny import App, render, ui, reactive
from pathlib import Path
import shinyswatch

# Import modules for plot rendering
import numpy as np
import matplotlib.pyplot as plt
import seaborn as sns
import pandas as pd

# Import modeling packages
from sklearn.neighbors import NearestNeighbors
from sklearn.preprocessing import StandardScaler

# import dictionary from local file (to save space)
import choices
d = choices.all_items
rest = choices.restaurants


# load Fast Food Data
infile = Path(__file__).parent / "fastfood_calories.csv"
data = pd.read_csv(infile)

# Features used for Nearest Neighbor Analysis
feats = ['calories', 'cal_fat', 'total_fat',
       'sat_fat', 'trans_fat', 'cholesterol', 'sodium', 'total_carb', 'fiber',
       'sugar', 'protein']

# Drop any rows with missing values
data.dropna(subset = feats, inplace = True)

# create Nearest Neighbor Model
val = NearestNeighbors(n_neighbors = 400)

# scale data
z = StandardScaler()
val.fit(z.fit_transform(data[feats]))

# fit Nearest Neighbor Model
distances, neighbors = val.kneighbors(z.fit_transform(data[feats]))


# UI
# this is where we tell shiny how things should look
app_ui = ui.page_fluid(
    # app title
    ui.h1("Fast Food Recommender", align = "center"),
    
    # sidebar
    ui.layout_sidebar(
        ui.panel_sidebar(
            # item to match
            ui.input_selectize("choice", "Select Item",
                               choices = d
                               ),
            # restaurants allowed                   
            ui.input_selectize("limits", "Restaurants Allowed",
                               choices = rest,
                                multiple = True,
                                selected = list(rest.keys()),
                            ),
            # number of recommendations
            ui.input_numeric("n", "Number of Recommendations", value = 3, min = 1, max = 50, step = 1),
            
        ),

        ui.panel_main(
           ui.h2("Recommendations"),
           ui.p("Recommendations are sorted by similarity to your selected item. Your item shown for comparison."),
           ui.output_table("table"),
        ),
    ),
    
)

# Server
# this is where we tell shiny how things should run
def server(input, output, session):
    # render table of recs
    @output
    @render.table
    def table():
        i = data.index[data['item'] == input.choice()] # get index of item
        recs = data.iloc[neighbors[i][0]] # get recs for item i
        
        recs = recs.loc[recs["restaurant"].isin(input.limits()),] # only allow listed restaurants 
        recs = recs.iloc[: , 1:] # get rid of extraneous column
        
        return recs.iloc[0:(input.n()+1)] # return n recommendations (+ self)
    
        

# Run our App
app = App(app_ui, server, debug=True)

More Shiny Features

We just used Shiny for Python to painlessly built a dynamic user interface to give fast food recommendations. But we haven’t even scratched the surface of Shiny’s power. For one, we can have dynamic text that updates based on a user’s selections (see the @render.text()s in the IRT app), or reactive effects that allow us to specify when updates to our output should happen (see @reactive.Effect in the Frog Jump Binomial App). Sharing your App

Last but not least, I want to talk about something we’ve been using all along: Shiny for Python’s serverless development. The Shiny for Python Examples Website is built to use Web Assembly (wasm) and Pyodide so that your apps that you build can run locally in the users browser rather than running on a server. This means you can share apps without setting up or monitoring a server. Shiny for Python’s website will encode the entire app into a URL that you can share with others (it is a very long URL though…) When they open that URL, the website will open the app and any associated data and run it locally on the user’s computer. I use this all the time in my classes by sharing a URL on Canvas so that students can do a class activity in the Shiny App without me hosting it on a server.

To share your app, Click the Share button at the top right corner of your page (right above the app). This will create two links: One to an Editor (which will open the app like we see it now, with the source code on the left and th app on the right) and one to an Application (which will open the app full screen in the browser). While the link is long, it’s all you need in order to share your app with others. For example, here’s our finished app!
